General style format and naming conventions

  • Class names use camel case after any layer prefix, for example .c-cardPreview.
  • Object classes are prefixed with o-.
  • Component classes are prefixed with c-.
  • State classes are prefixed with is-.
  • Use // for short Sass comments and reserve longer comments for behaviour that is not obvious.
  • Use semantic Sass variables for reusable colour decisions. Raw hex and rgba values belong in token files or rare one-off implementation details.

Example:

{% highlight scss %} // Reusable component styles use a c- prefix. .c-cardPreview { color: components.$card-preview-color; background-color: components.$card-preview-bg; &-media { display: block; } &--compact { padding-block: vertical-rhythm.vr-space(0.5); } } {% endhighlight %}

File structure

Library Sass should stay grouped by responsibility. The main entrypoint should compose the public library layers in a predictable order.

{% highlight html %} sass +-- base | +-- functions | | +-- _functions.scss | +-- mixins | | +-- _breakpoints.scss | +-- _colours.scss | +-- _components.scss | +-- _ground-zero.scss | +-- _normalize.scss | +-- _typography.scss +-- components | +-- _buttons.scss +-- objects | +-- _lists.scss +-- utilities | +-- _grid.scss +-- style.scss {% endhighlight %}
  • Use Sass @use rather than @import.
  • Keep design tokens and mixins in base.
  • Keep public reusable selectors in objects, components and utilities.
  • Create or extend a documented reusable pattern rather than adding unrelated selectors to the entrypoint.

Base

Base styles define global element defaults, design tokens, functions and mixins. Element selectors belong here when they describe the default behaviour of the library.

A Base rule is applied to an element using an element selector, a descendent selector, or a child selector, along with any pseudo-classes. It doesn’t include any class or ID selectors. It is defining the default styling for how that element should look in all occurrences on the page.

—SMACSS
{% highlight scss %} html { font-size: 62.5%; } body { @include font.font-size; @include vertical-rhythm.vr-line-height; font-family: typography.$base-font-family; color: colours.$base-color; background-color: colours.$base-background-color; } {% endhighlight %}

Tokens

Token files expose reusable values for colours, typography, grid settings and component decisions.

  • Use semantic token names for decisions made by the library.
  • Keep raw palette names available only when they are intentionally part of the documented API.
  • Use !default for values that consuming projects may override.

Mixins

Mixins are reusable Sass helpers that do not output CSS unless included by another rule.

Naming convention

  • Mixins use kebab case names.
  • Mixin arguments should describe the design decision, not the implementation detail.
{% highlight scss %} @mixin visually-hidden() { position: absolute !important; overflow: hidden !important; width: 1px !important; height: 1px !important; padding: 0 !important; border: 0 !important; clip: rect(1px, 1px, 1px, 1px) !important; } .visuallyHidden { @include typography.visually-hidden; } {% endhighlight %}

Objects

Objects are reusable structural abstractions. They define layout or behaviour for a repeatable pattern without carrying project-specific visual design.

Naming convention

Objects use BEM style block, element and modifier notation.

  • An object must have a unique CSS class name.
  • An object name must be prefixed with o-.
  • An object must use a camel case name.
  • Different behaviours of an object must be added as modifiers.
  • Objects should fill their container rather than owning fixed width or height.

Example:

{% highlight scss %} .o-listInline { > li { display: inline-block; } } .o-listInline--spaced { > li + li { margin-left: vertical-rhythm.vr-space(0.5); } } {% endhighlight %}

Components

Components are reusable interface modules. They can use base tokens, mixins, objects and utilities, but their styling should remain scoped to the component contract.

Naming convention

Components use BEM style block, element and modifier notation.

  • A component must have a unique CSS class name.
  • A component name must be prefixed with c-.
  • A component must use a camel case name.
  • Component variants should be modifiers rather than separate unrelated classes.
  • A component should not depend on a specific application template to make sense.

Example:

{% highlight scss %} .c-card { padding: vertical-rhythm.vr-space(); border: 1px solid components.$card-border-color; &-title { @include typography.h3; } &--featured { border-color: components.$card-featured-border-color; } } {% endhighlight %}

Utilities

Utilities are small, single-purpose classes that make composition easier without creating a new component.

Global classes

  • Utilities should do one thing and have predictable specificity.
  • Use utilities for small composition adjustments, not as a substitute for component styling.
  • Prefer component variables and component classes when a visual decision belongs to a reusable pattern.
  • Review broad layout helpers before treating them as reusable utilities.
{% highlight css %} .h1 { font-size: 60px; } .block { display: block; } .textCenter { text-align: center; } .m-b--full { margin-bottom: 24px; } {% endhighlight %}

State

State classes represent temporary UI state such as hidden, active, selected or expanded. They may be shared by CSS and JavaScript.

  • State classes are prefixed is-.
  • Use !important in state classes only when the state must override component styles.
  • Do not use state classes for permanent variants; use component modifiers instead.

Example:

{% highlight css %} .is-hidden { display: none; } .is-active { font-weight: 700; } {% endhighlight %}
